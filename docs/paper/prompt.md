# Prompt

This project is made for the BioASQ 14B challenge. Now, it's time to write the paper at @docs/paper . We've been participating in the challenge for the past years, so each year the corresponding paper reflects on the work done, compared to the previous year. We also need to explain how it has improved our results.

The 13B paper is available at @paper_13b.pdf for reference on the structure and content.

## What was done this year

General:

- Huge code refactor and restructuring, with a focus on modularity

Phase A (retrieval):

- Switch from PyTerrier PISA to pg_textsearch for BM25 retreival
  - Faster retrieval
  - Integration with PostgreSQL, which means it serves as a single source of truth for all the PubMed documents
  - No need to load the entire indexes into memory at each pipeline iteration, which takes disk I/O time
- Indexing of PubMed document embeddings using Qdrant
  - No need to precompute and store all cross-embedding similarities, which is a huge improvement in terms of storage and flexibility
  - GPU acceleration for similarity search, which is much faster than CPU-based approaches
  - More efficient storage, instead of storing plain Numpy arrays
- Use of Wayback Machine to retrieve the PubMed documents, which enables to go back to 2017 and get a broader set of documents
- Usage of Text Embeddings Interface (TEI) from Hugging Face for embedding generation, which provides a fast, Rust-based implementation
- New reranker training pipeline
- RRF and weighted sum for fusion of BM25 and dense retrieval scores (with special attention to RRF)
- HyDE: replace the original query with a hypothetical answer generated by the LLM, which is then used to retrieve more relevant documents
- Context1: https://www.trychroma.com/research/context-1 LLM for retrieval, which uses the standard RAG pipeline to improve retrieval results
- Snippets generation: a category in BioASQ that we were not participating in before
- Include dense retrieval in negative generation for the reranker training (was BM25 only before)
- Future work: SPLARe, ColBERT

Phase A+/B (generation):

- LLM as a judge: instead of a simple ensemble of different models with majority voting, use an LLM to evaluate the outputs of different models and choose the best one, or even combine them in a more intelligent way
- Agent quorum:
  - Multiple agents with different prompts and/or models
  - Documents and corresponding snippets are given to all agents, in an adaptive retention way. Three random documents are given to each, and each agent chooses what documents it wants to keep. If it chooses less than three, the next round will have the documents it keeps + other randoms to make it three again. If it keeps the three of them, the next round will have the three + 1 random document, and so on. This way, the agents can focus on the documents they find more relevant, while still having some diversity in the documents they see.
  - They debate each other and output their opinion and a score of ["strongly_disagree", "disagree", "agree", "strongly_agree"]
  - Multiple rounds (all agents must output their opinion and score before the next round starts, in random order), until they all reach a consensus (e.g., all strongly agree), reach a maximum number of rounds or they keep the same score for a certain number of rounds
  - During their debate, they comment on other agents' opinions and scores, which can lead to score changes
  - In the end, let one model summarize the outputs of all the agents into one final answer
- Use snippets for better generation: snippets have found to make the agent quorum faster, as they focus more on the relevant parts of the documents, which makes it easier for them to reach a consensus

## Paper structure

- Introduction
- Previous work (summary of 13B modifications)
- Methodology (changes for this year)
- Results (ours and official ones)
  - For our internal results, go to https://huggingface.co/IEETA/BioASQ-14B/tree/main and, for each folder, check the ranx_results.json file, if exists.
  - For official results, check the @results.md file.
- Discussion (what we learned, future work, etc.)
- Conclusion

## Language guidelines

Write a formal, IEEE-like English, with a clear and concise style. Use technical terms appropriately, and ensure that the paper is well-structured and easy to follow. Avoid typical AI-generated text patterns, such as bullet points or bolded text, and instead use full sentences and paragraphs to convey information.
